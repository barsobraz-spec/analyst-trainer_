# PRD: Улучшение учебной платформы Analyst Trainer

## Статус

Draft.

Документ описывает техническое и продуктово-UX улучшение существующего `analyst-trainer` без переписывания проекта с нуля.

## Цель

Сделать Analyst Trainer более удобной, чистой и масштабируемой учебной платформой для Data Analyst, чтобы позже можно было безопасно добавлять новые месяцы, недели, дни, задачи, практику, проверку результатов и персональный прогресс.

Главная цель этого PRD - подготовить систему к росту контента и функциональности, не занимаясь ручным заполнением учебного контента месяцев 2-7.

## Важное ограничение по контенту

Заполнение подробных учебных дней, блоков, задач, практики, контрольных вопросов и типичных ошибок для месяцев 2-7 не входит в этот скоуп.

Разработческие работы должны подготовить структуру, UI и валидаторы так, чтобы владелец проекта мог позже сам заполнить месяцы 2-7.

Допустимо:

- добавлять optional-поля схемы;
- добавлять валидаторы и предупреждения;
- добавлять UI для пустого или чернового контента;
- добавлять документацию по заполнению;
- поддерживать текущие данные месяца 1.

Недопустимо:

- генерировать или дописывать подробный контент месяцев 2-7;
- удалять существующие данные;
- менять существующие ID;
- ломать текущие маршруты;
- переписывать приложение с нуля.

## Контекст

Проект уже содержит:

- статическое приложение без сборки;
- hash-router;
- левую навигацию и мобильную нижнюю навигацию;
- учебный план в `learning-plan/data/plan.json`;
- задачи в `learning-plan/data/tasks.json`;
- практику в `learning-plan/data/practiceContent.json`;
- проекты и карьерный трек;
- кейсы тренажера в `cases/`;
- пользовательский прогресс в IndexedDB;
- валидатор учебных данных.

Текущая система уже работоспособна, но при росте контента появятся риски:

- неявные связи между днем, задачей, практикой и кейсом;
- монолитные JSON;
- большой CSS-файл;
- крупные DOM-компоненты;
- слабая масштабируемость экрана "Практика";
- перегрузка длинных страниц;
- неполная проверка связей данных.

## Целевая аудитория

Основной пользователь - человек, который самостоятельно проходит маршрут Data Analyst и хочет:

- понимать, что делать сегодня;
- видеть путь месяц -> неделя -> день;
- быстро переходить к задачам и практике;
- отслеживать прогресс;
- возвращаться к слабым местам;
- собирать портфолио;
- позже получать проверку результатов.

Второй пользователь - владелец/автор контента, который будет добавлять месяцы, недели, дни, задачи и практику в JSON.

## Non-goals

В этот PRD не входят:

- заполнение подробного контента месяцев 2-7;
- полный редизайн визуального языка;
- миграция на React/Vue/Svelte;
- backend и авторизация;
- полноценная ИИ-проверка с внешним API;
- удаление текущих JSON;
- изменение существующих маршрутов;
- изменение существующих ID кейсов, задач, модулей и проектов.

## Принципы

1. Маленькие обратимые изменения.
2. Данные отдельно от UI.
3. UI не должен угадывать связи, если их можно описать в данных.
4. Старые ID сохраняются.
5. Новые поля добавляются как optional до полной миграции.
6. Один плохой элемент контента не должен ломать приложение.
7. Пустой контент месяцев 2-7 должен отображаться как нормальное состояние, а не как ошибка.

## Продуктовые требования

### Главный экран

Главный экран должен оставаться рабочим столом, а не лендингом.

Нужно:

- показывать следующий понятный шаг;
- различать "продолжить кейс" и "продолжить учебный день";
- давать быстрый вход в "Сегодня", "План", "Задачи", "Практику", "Проекты";
- не перегружать пользователя метриками;
- корректно работать без заданной даты старта.

### Раздел "Сегодня"

Раздел должен быть ежедневным центром работы.

Нужно:

- показывать текущий день по дате старта;
- показывать тему дня;
- показывать связанные задачи;
- показывать связанные практики, если они явно указаны;
- если связей нет, использовать безопасный fallback;
- поддерживать минимальный и обычный режим дня;
- показывать дневник аналитика;
- не ломаться, если в месяце 2-7 нет подробных `days`.

### Раздел "План"

Раздел должен объяснять структуру месяц -> неделя -> день.

Нужно:

- сохранить текущие deep-links `#/learning/plan?month=...&week=...&day=...`;
- сделать переходы между днями очевидными;
- показывать пустые или неполные дни как "контент будет заполнен позже";
- не требовать полного заполнения месяцев 2-7;
- вынести сложные части UI в отдельные модули.

### Раздел "Задачи"

Раздел должен быть рабочим чек-листом, а не длинной стеной.

Нужно:

- группировать задачи по месяцу, навыку и статусу;
- сохранять фильтры;
- показывать наличие практики у задачи;
- давать быстрый переход к связанной практике и модулям тренажера;
- улучшить мобильный сценарий, где много одинаковых select-статусов.

### Раздел "Практика"

Текущий экран со случайным кейсом нужно развить в practice hub.

Нужно:

- оставить быстрый случайный кейс;
- добавить фильтры по модулю, сложности, статусу и навыку;
- добавить режим "по слабым местам";
- добавить режим "по задачам учебного плана";
- показывать, какая практика связана с какой задачей;
- поддержать будущие `practiceId`.

### UI и адаптивность

Нужно:

- исправить нижнюю навигацию, чтобы она не перекрывала контент на desktop;
- добавить безопасный нижний отступ на mobile;
- уменьшить вложенность карточек;
- улучшить сканируемость длинных страниц;
- сохранить текущий визуальный стиль без полного редизайна.

## Технические требования

### Архитектура UI

Нужно разрезать крупные файлы без изменения поведения.

Кандидаты:

- `modules/learning/PlanView.js`;
- `modules/learning/TasksView.js`;
- `modules/practice/PracticeView.js`;
- `styles.css`.

Целевое разбиение:

```text
modules/learning/plan/
  PlanView.js
  PlanNavigator.js
  DayDetail.js
  MonthPanel.js
  MonthlyExamPanel.js

modules/learning/tasks/
  TasksView.js
  TaskFilters.js
  TaskGroup.js
  TaskPracticeDetails.js

modules/practice/
  PracticeView.js
  PracticeFilters.js
  PracticeCard.js
  PracticePicker.js

styles/
  tokens.css
  base.css
  layout.css
  learning.css
  practice.css
  cases.css
```

Разбиение CSS можно делать постепенно. На первом шаге допустимо оставить один CSS, но новые секции должны быть четко сгруппированы.

### Данные

Существующие поля сохраняются.

Новые поля добавляются как optional:

```json
{
  "day": {
    "id": "day-01-01-01",
    "month": 1,
    "week": 1,
    "day": 1,
    "taskIds": ["task-001"],
    "practiceIds": ["practice-task-001"],
    "caseIds": ["dashboard-001"]
  }
}
```

Для практики:

```json
{
  "id": "practice-task-001",
  "taskId": "task-001",
  "status": "ready",
  "autoCheck": {
    "type": "manual"
  },
  "rubric": []
}
```

Для месяцев 2-7 эти поля могут отсутствовать или быть пустыми до ручного заполнения.

### Валидация

Валидатор должен проверять:

- JSON читается;
- существующие ID не дублируются;
- все `taskId` в practice существуют;
- все `trainerModules` известны;
- все optional `taskIds`, `practiceIds`, `caseIds` ссылаются на существующие элементы;
- месяцы 2-7 могут иметь неполные `days`, но это должно быть warning, а не hard error;
- месяц 1 должен продолжать проходить текущие проверки.

### Загрузка данных

Текущий `loadLearningContent()` может остаться на первом этапе.

Позже нужно подготовить:

- легкий индекс задач;
- легкий индекс практики;
- загрузку detail-практики по `taskId` или `practiceId`;
- возможность разнести практику по файлам без изменения UI.

### IndexedDB

Не менять существующие stores без отдельного migration PRD.

Можно добавлять пользовательское состояние только через существующие или новые миграции, если это требуется для UI.

Для этого PRD основная работа должна быть в контентной схеме, UI и валидаторах.

## Подготовка под будущую ИИ-проверку

В этот скоуп не входит подключение ИИ.

Нужно только подготовить схему:

```json
{
  "submissionSchema": {
    "type": "text",
    "fields": ["answer", "sql", "businessConclusion"]
  },
  "rubric": [
    {
      "id": "business-context",
      "title": "Понимание бизнес-контекста",
      "maxScore": 3
    }
  ],
  "autoCheck": {
    "type": "manual",
    "futureTypes": ["sql_result", "table_shape", "ai_rubric"]
  }
}
```

Для месяцев 2-7 эти поля не заполняются автоматически.

## Acceptance criteria

PRD считается реализованным, если:

- текущие маршруты открываются без ошибок;
- нижняя навигация не перекрывает desktop-контент;
- мобильные экраны имеют безопасный нижний отступ;
- `node analyst-trainer/learning-plan/validate-data.mjs` проходит;
- валидатор умеет проверять новые optional-связи;
- `PlanView.js` уменьшен за счет вынесения компонентов;
- `PracticeView` поддерживает хотя бы базовые фильтры;
- существующие ID не изменены;
- месяцы 2-7 не заполнялись новым подробным контентом;
- пустой или неполный контент месяцев 2-7 отображается корректно.

## Этапы реализации

### Этап 1. Безопасные UX-фиксы и валидаторы

Цель: убрать явные проблемы без изменения модели данных.

Работы:

- исправить поведение нижней навигации;
- добавить mobile bottom padding;
- усилить `validate-data.mjs`;
- добавить warnings для неполных месяцев 2-7;
- убрать или централизовать ручные query suffix в импортах.

Риск: низкий.

Проверка:

```bash
node analyst-trainer/learning-plan/validate-data.mjs
```

И вручную открыть:

- `#/`;
- `#/learning/today`;
- `#/learning/plan`;
- `#/learning/plan?month=1&week=1&day=1`;
- `#/learning/tasks?skill=excel`;
- `#/practice`.

### Этап 2. Разрезать крупные UI-файлы

Цель: сделать структуру поддерживаемой.

Работы:

- вынести `DayDetail`;
- вынести `PlanNavigator`;
- вынести `MonthlyExamPanel`;
- вынести элементы задач;
- сохранить поведение и маршруты.

Риск: средний.

### Этап 3. Явные связи данных

Цель: подготовить данные к росту без заполнения месяцев 2-7.

Работы:

- добавить поддержку optional `day.id`;
- добавить поддержку optional `taskIds`, `practiceIds`, `caseIds`;
- добавить поддержку optional `practice.id`;
- обновить нормализацию данных;
- добавить fallback, если связей нет.

Риск: средний.

### Этап 4. Practice Hub

Цель: сделать практику полезной не только как random кейс.

Работы:

- добавить фильтры;
- добавить режим "слабые места";
- добавить режим "практика по задачам";
- сохранить кнопку случайного кейса.

Риск: средний.

### Этап 5. Подготовка под проверку результатов

Цель: добавить схему для будущей проверки без подключения внешнего ИИ.

Работы:

- добавить optional `rubric`;
- добавить optional `submissionSchema`;
- добавить optional `autoCheck`;
- добавить UI-заглушки для критериев проверки.

Риск: низкий.

### Этап 6. Подготовка к разбиению JSON

Цель: убрать риск тяжелого `practiceContent.json` при росте.

Работы:

- описать manifest для practice;
- подготовить loader API;
- оставить текущий файл совместимым;
- не переносить весь контент сразу.

Риск: средний.

## Что не делать в ближайших работах

- Не заполнять месяцы 2-7 подробными днями.
- Не менять `task-001`, `dashboard-001`, `detective-001` и другие существующие ID.
- Не удалять текущие поля JSON.
- Не переписывать router.
- Не менять IndexedDB схему без отдельной причины.
- Не делать новый дизайн ради дизайна.

## Первые 5 действий

1. Исправить desktop/mobile поведение нижней навигации.
2. Добавить safe bottom spacing для мобильных экранов.
3. Добавить warnings в валидатор для неполных месяцев 2-7.
4. Вынести `DayDetail` из `PlanView.js`.
5. Добавить поддержку optional `day.id`, `taskIds`, `practiceIds`, `caseIds` без заполнения месяцев 2-7.
