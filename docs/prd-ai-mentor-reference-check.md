# PRD: AI-ментор, связи тем и проверка по эталону

## Статус

Draft.

Документ описывает развитие Analyst Trainer в сторону персонального AI-ментора для подготовки Data Analyst. Цель - связать темы, задачи, модули, кейсы и эталоны в единую учебную траекторию, а AI использовать как наставника, который проверяет ответ ученика по эталону, объясняет ошибки и предлагает следующий шаг.

## Коротко

Сейчас в приложении уже есть:

- учебный план и задачи;
- модули тренажера 5.1-5.8;
- кейсы с эталонами;
- SQL-автопроверка по результату запроса;
- самооценка для текстовых кейсов;
- базовая настройка AI-proxy в `#/settings`.

Нужно добавить слой:

```text
тема -> задачи -> модуль -> кейсы -> эталон -> AI-проверка -> что повторить дальше
```

AI должен быть не "оракулом с готовым ответом", а ментором:

1. оценивает по эталону;
2. показывает, что сделано правильно;
3. находит ошибку или пробел;
4. задает наводящий вопрос;
5. предлагает, что повторить;
6. только при выбранном режиме может показать полный эталон.

## Цели

### Главная цель

Сделать так, чтобы пользователь мог решать кейсы и получать качественную обратную связь как от хорошего наставника: не просто "правильно/неправильно", а почему, где логическая ошибка и какую тему повторить.

### Продуктовые цели

- связать учебные темы с модулями, кейсами, задачами и проектами;
- добавить AI-проверку по эталону во все модули, где это уместно;
- сохранить автопроверку там, где она точнее AI;
- сделать AI-ментора безопасным: ключ DeepSeek не хранится во фронтенде;
- не ломать офлайн-режим: без AI приложение продолжает работать;
- дать пользователю понятный следующий шаг после ошибки.

## Non-goals

В этот скоуп не входит:

- замена всех проверок на AI;
- хранение DeepSeek API key в браузере;
- обязательная авторизация пользователей;
- переписывание приложения на React/Vue;
- генерация всех эталонов заново;
- автоматическая отправка личных данных пользователя во внешний API;
- проверка без подтвержденного контекста и эталона;
- сохранение полной истории AI-проверок в MVP;
- проверка README/портфолио в MVP.

## Принципы

1. **Эталон первичен.** AI проверяет относительно эталона, критериев и контекста кейса.
2. **Автопроверка важнее AI там, где она возможна.** SQL-результат и числовые ответы сначала проверяются алгоритмом.
3. **AI объясняет, а не просто ставит оценку.**
4. **AI не должен сразу раскрывать готовое решение в режиме подсказки.**
5. **Без proxy AI-функции отключены, но приложение работает.**
6. **Все связи должны жить в данных, а не быть зашиты в UI.**
7. **Один плохой AI-ответ не должен записывать прогресс автоматически.**

## Пользовательские сценарии

### Сценарий 1: Проверка ответа по эталону

Пользователь решает текстовый кейс, пишет вывод и нажимает "Проверить с AI-ментором".

AI получает:

- модуль;
- тему;
- условие кейса;
- ответ пользователя;
- эталонный разбор;
- критерии оценки;
- связанные задачи и навыки.

AI возвращает:

- оценку 0-100;
- что правильно;
- что неполно или неверно;
- наводящий вопрос;
- 2-3 темы для повторения;
- ссылки на связанные задачи/кейсы.

### Сценарий 2: Подсказка без раскрытия ответа

Пользователь застрял и выбирает режим "Дай подсказку, но не ответ".

AI получает условие, тему, rubric и commonMistakes, но не получает полный `reference`. Ответ AI:

- указывает, на какую часть условия смотреть;
- задает вопрос;
- предлагает проверить одно предположение;
- не раскрывает готовый вывод.

### Сценарий 3: SQL с автопроверкой и AI-объяснением

Пользователь пишет SQL. Система сначала сравнивает результат с эталоном.

Если результат неверный:

- автопроверка сообщает, что результат не совпал;
- AI получает SQL пользователя, эталонный SQL, схему таблиц, формулировку задачи;
- AI объясняет вероятную логическую ошибку: JOIN, GROUP BY, NULL, фильтр, гранулярность.

AI не должен заменять сравнение результата.

### Сценарий 4: AI как ментор по теме

На странице темы пользователь видит блок "Связанные темы и следующий шаг".

Например:

```text
Текущая тема: SQL JOIN
До этого: SELECT, WHERE
Связанные модули: 5.5 SQL Investigation, 5.1 Data Detective
Кейсы: sql-004, sql-005
Проект: продуктовая воронка
Если ошибся: повторить гранулярность и дубли при JOIN
```

AI может объяснить, почему эта тема важна и какие задачи лучше решить дальше.

### Сценарий 5: Проверка проекта и README

Пользователь вставляет README или описание проекта. AI проверяет:

- есть ли бизнес-вопрос;
- понятна ли методология;
- есть ли выводы с цифрами;
- есть ли рекомендации;
- выглядит ли проект как рабочая задача, а не учебная тетрадка;
- что улучшить перед отправкой работодателю.

## Режимы AI-ментора

Минимальный набор режимов:

| Режим | Назначение |
| --- | --- |
| Проверить по эталону | Оценить ответ относительно reference/rubric |
| Дать подсказку | Помочь без раскрытия готового ответа |
| Объяснить ошибку | Разобрать, почему ответ неверный |
| Проверить SQL | Объяснить ошибку SQL после автопроверки |
| Проверить вывод для бизнеса | Проверить, есть ли управленческий вывод |
| Проверить README | Улучшить портфолио-проект |
| Mock-интервью | Задать вопросы и оценить ответ |
| Что повторить дальше | Дать следующий шаг по слабым местам |

## Проверка по модулям

### 5.1 Data Detective

AI проверяет:

- точность вывода;
- полноту наблюдений;
- отсутствие ложной причинности;
- связь вывода с таблицей/фактами;
- качество финального бизнес-ответа.

Контекст:

- `payload.scenario`;
- `payload.table`;
- `payload.metrics`;
- `payload.question`;
- `payload.reference`;
- ответ пользователя;
- цепочка рассуждений.

### 5.2 Hypothesis Trainer

AI проверяет:

- качество гипотезы;
- проверяемость;
- связь с метриками;
- приоритизацию;
- риски и guardrail-метрики.

### 5.3 Dashboard Analysis

AI проверяет:

- правильность чтения графика;
- выбор главного инсайта;
- качество бизнес-вывода;
- перегибы в интерпретации;
- рекомендации по визуализации.

### 5.4 Root Cause Analysis

AI проверяет:

- глубину 5 Why;
- отличие симптома от причины;
- корректность cause tree/fishbone;
- наличие подтверждений;
- качество корневой причины.

### 5.5 SQL Investigation

AI не заменяет автопроверку. AI объясняет:

- почему результат не совпал;
- где ошибка в JOIN/GROUP BY/WHERE/HAVING/window;
- как проверить гранулярность;
- как найти дубли;
- как безопасно исправить запрос.

Контекст:

- формулировка подзадачи;
- SQL пользователя;
- эталонный SQL;
- схема таблиц;
- сообщение автопроверки.

### 5.6 Business Simulator

AI проверяет:

- качество бизнес-решения;
- учет trade-offs;
- связь решения с метриками;
- риски;
- альтернативы;
- качество объяснения для руководителя.

### 5.7 Automation Designer

AI проверяет:

- процесс;
- входы/выходы шагов;
- пропущенные проверки;
- измеримый результат;
- риски автоматизации.

### 5.8 Learning Analytics

AI использует агрегированный прогресс:

- слабые модули;
- задачи со статусом `repeat`;
- низкие score;
- незавершенные проекты;
- историю активности.

AI предлагает:

- что повторить сегодня;
- какие 2-3 кейса решить;
- какую тему закрыть перед следующим проектом.

Нужно добавить отдельный слой данных для связей.

Рекомендуемый файл:

```text
learning-plan/data/topicGraph.json
```

Пример:

```json
{
  "schemaVersion": 1,
  "topics": [
    {
      "id": "sql-joins",
      "title": "SQL JOIN",
      "skill": "sql",
      "month": 2,
      "prerequisites": ["sql-select", "sql-where"],
      "next": ["sql-aggregation", "sql-window-functions"],
      "related": ["data-quality", "granularity"],
      "taskRefs": ["task-028", "task-029", "task-030"],
      "caseRefs": ["sql-004", "sql-005"],
      "moduleRefs": ["5.5", "5.1"],
      "projectRefs": ["product-funnel-retention"],
      "commonMistakes": [
        "many-to-many JOIN завышает метрики",
        "фильтр в WHERE превращает LEFT JOIN в INNER JOIN",
        "агрегация сделана на неправильной гранулярности"
      ]
    }
  ]
}
```

### Где показывать связи

В разделе "Задачи":

- текущая тема;
- prerequisite;
- следующая тема;
- связанные модули;
- связанные кейсы.

В кейсе:

- тема кейса;
- связанные задачи;
- что повторить при ошибке.

В AI-менторе:

- AI получает topic graph как контекст и возвращает следующий шаг.

## API proxy

Фронтенд не должен обращаться к DeepSeek напрямую с ключом.

Нужен endpoint:

```text
POST /api/ai-review
```

Запрос:

```json
{
  "provider": "deepseek",
  "model": "deepseek-v4-pro",
  "mode": "reference_check",
  "context": {
    "module": "5.1",
    "topic": "funnel-analysis",
    "case": {},
    "reference": {},
    "rubric": [],
    "related": {}
  },
  "studentAnswer": "..."
}
```

Ответ:

```json
{
  "score": 74,
  "verdict": "partially_correct",
  "strengths": [
    "Правильно заметил падение на шаге корзины"
  ],
  "issues": [
    "Причина названа без проверки альтернатив"
  ],
  "mentorQuestion": "Как ты отличишь проблему трафика от проблемы UX на шаге оплаты?",
  "improvements": [
    "Сравни конверсию по устройствам",
    "Проверь изменение источников трафика"
  ],
  "nextSteps": [
    {
      "type": "task",
      "id": "task-106",
      "title": "Анализ воронки"
    }
  ]
}
```

Ошибки:

- `401`: proxy не настроен или ключ неверный;
- `429`: лимит запросов;
- `502`: модель недоступна;
- `400`: пустой ответ пользователя или нет контекста.

Требования к proxy:

- proxy живет в отдельном Vercel project/serverless endpoint;
- frontend хранит только URL proxy в `#/settings`;
- DeepSeek API key хранится только в server-side env;
- proxy применяет backend rate limit: 20 запросов в день и 5 запросов в минуту на IP или anonymous session;
- proxy должен возвращать структурированный JSON по контракту выше;
- frontend/backend валидируют ответ AI перед показом;
- при невалидном JSON или неполном ответе UI показывает безопасную ошибку и не меняет прогресс;
- CORS разрешен только для домена приложения и локального dev-окружения.

## Безопасность

Нельзя:

- хранить DeepSeek API key в `localStorage`;
- класть ключ в `config.js`;
- коммитить `.env`;
- отправлять в AI персональные данные без явного действия пользователя;
- автоматически сохранять AI-score как прогресс прохождения.

Нужно:

- предупреждать, что ответ будет отправлен во внешний AI proxy;
- перед первой AI-проверкой показывать consent warning и сохранять локальный consent flag;
- дать пользователю ссылку или раскрываемый блок "что отправляется";
- показывать краткое превью состава данных перед отправкой: кейс, модуль, тема, ответ пользователя, reference/rubric;
- позволить отключить AI;
- показывать ошибку, если proxy не настроен;
- логировать только технический статус, не полный ответ пользователя.

## UX

### В кейсе

Блок:

```text
AI-ментор

Режим: Проверить по эталону
Тема: SQL JOIN

[Проверить мой ответ]
[Дай подсказку]
[Что повторить?]
```

Результат:

```text
Оценка: 74/100

Что хорошо:
- ...

Что исправить:
- ...

Вопрос ментора:
...

Следующий шаг:
- task-106 Анализ воронки
- sql-005 JOIN и агрегации

Действия пользователя:
- Отметить как исправлено
- Повторить позже
```

AI-feedback не меняет прогресс автоматически. Пользователь сам выбирает действие после разбора.

### В теме

Блок:

```text
Связанные темы
До этого: SELECT, WHERE
Дальше: GROUP BY, оконные функции
Модули: 5.5, 5.1
Кейсы: sql-004, sql-005
```

### В настройках

Текущий блок `AI-проверка` должен стать настройкой:

- AI-proxy URL;
- модель: по умолчанию `deepseek-v4-pro`;
- тестовый запрос;
- статус подключения.

Если proxy поддерживает выбор модели по режимам, используются:

- `deepseek-v4-pro` для `reference_check`, `explain_error`, `sql_review`;
- `deepseek-v4-flash` для `hint` и `next_step`.

Основная работа пользователя должна быть не в настройках, а в кейсах и темах.

## Архитектура

Новые модули:

```text
core/aiReview.js              # клиент AI-proxy
core/mentorContext.js         # сбор контекста кейса/темы/эталона
core/topicGraph.js            # загрузка и валидация связей тем

```

Возможный backend:

```text
api/ai-review.ts              # Vercel serverless proxy
```

Если текущий проект остается полностью статическим, proxy должен жить в отдельном репозитории или Vercel-проекте.

## Хранение истории AI-проверок

Не входит в MVP. В MVP показывается только текущий результат AI-проверки без сохранения полного текста ответа пользователя и AI-feedback.

После MVP можно добавить локальную историю:

```text
aiMentorReviews
```

Поля:

```json
{
  "reviewId": "review-...",
  "caseId": "detective-001",
  "module": "5.1",
  "topicId": "funnel-analysis",
  "mode": "reference_check",
  "score": 74,
  "summary": "Ответ частично верный...",
  "createdAt": 1710000000000
}
```

История, если будет добавлена позже, должна иметь кнопку удаления и не должна автоматически отправляться во внешний API.

## Этапы внедрения

### Этап 1: Данные связей

- создать `topicGraph.json`;
- добавить загрузчик `core/topicGraph.js`;
- добавить валидацию ссылок на задачи/кейсы/модули;
- показать связи на экране "Задачи".

Критерий готовности:

- тема показывает prerequisites, next, related, moduleRefs, caseRefs;
- битая ссылка не ломает экран, а валидатор предупреждает.

### Этап 2: Контекст AI-ментора

- создать `core/mentorContext.js`;
- собрать контекст для 5.1 и 5.5;
- нормализовать `reference`;
- добавить режимы `hint`, `reference_check`, `explain_error`, `next_step`.

Критерий готовности:

- для `detective-001` контекст содержит scenario/question/reference/studentAnswer;
- для SQL-кейса контекст содержит prompt/userSql/referenceSql/schema.

### Этап 3: UI AI-ментора в кейсе

- создать `core/components/AiMentor.js`;
- встроить в 5.1 и 5.5;
- использовать настройки из `core/aiReview.js`;
- показывать ошибку, если proxy не настроен;
- показать consent warning перед первым AI-запросом;
- показать краткое превью состава данных перед отправкой;
- валидировать структурированный ответ AI;
- не менять прогресс автоматически после AI-feedback.

Критерий готовности:

- кнопка AI-ментора есть в кейсе;
- без proxy выводится понятное сообщение;
- с тестовым proxy отображается score и feedback;
- невалидный ответ AI не ломает UI;
- пользователь сам выбирает "Отметить как исправлено" или "Повторить позже".

### Этап 4: Расширение на остальные модули

- 5.2, 5.3, 5.4, 5.6, 5.7;
- добавить module-specific rubric;
- добавить подсказки "что повторить".

### Этап 5: AI-ментор по слабым местам

- интеграция с 5.8 Learning Analytics;
- AI анализирует события, weak spots, задачи `repeat`;
- предлагает план на 1-3 дня.

## Acceptance Criteria MVP

MVP считается готовым, если:

- есть `topicGraph.json` минимум для 10 ключевых тем;
- экран задач показывает связи темы;
- в 5.1 Data Detective есть AI-ментор;
- в 5.5 SQL Investigation AI объясняет ошибку после неверной автопроверки;
- без AI-proxy приложение работает как раньше;
- AI-proxy URL настраивается в `#/settings`;
- DeepSeek key не хранится во frontend-коде;
- proxy имеет server-side rate limit;
- перед первым AI-запросом есть consent warning;
- перед отправкой доступно краткое превью состава данных;
- режим `hint` не отправляет полный `reference`;
- ответ AI валидируется перед показом;
- AI-feedback не меняет прогресс без явного действия пользователя;
- smoke-тесты роутов проходят;
- валидация данных предупреждает о битых topic/task/case/module refs.

## Метрики успеха

- пользователь чаще исправляет ответ до просмотра эталона;
- растет доля повторно решенных слабых кейсов;
- снижается число задач со статусом `repeat`;
- проекты получают более полные README;
- пользователь может объяснить ошибку своими словами после AI-разбора.

## Решения после ревью

- История AI-проверок не входит в MVP.
- Proxy живет в отдельном Vercel project/serverless endpoint.
- По умолчанию используется `deepseek-v4-pro`; для `hint` и `next_step` можно использовать `deepseek-v4-flash`, если proxy поддерживает выбор модели.
- Backend rate limit входит в MVP.
- Перед первым запросом нужен consent warning.
- Перед отправкой нужно показывать краткое превью состава данных.
- README/портфолио review остается за пределами MVP.

## Риски

- AI может дать уверенный, но неверный разбор.
- Пользователь может начать копировать ответы AI вместо обучения.
- Если backend rate limit реализован неверно, proxy может быстро потратить лимит.
- Если отправлять слишком большой контекст, ответы будут дорогими и медленными.
- Если не сделать topic graph, AI не сможет давать точный следующий шаг.

## Рекомендация по продуктовой логике

AI-ментор должен быть встроен не как "чат рядом", а как учебный цикл:

```text
попытка -> подсказка -> исправление -> проверка по эталону -> объяснение ошибки -> следующий шаг
```

Тогда AI усиливает тренажер, а не превращает его в генератор готовых ответов.

## Дополнение после ревью

### Уточнения, учтенные в ТЗ

1. **Proxy и лимиты входят в MVP.** Proxy должен жить отдельно от статического frontend, хранить ключ только server-side и иметь backend rate limit.
2. **Контракт ответа AI строгий.** Ответ валидируется перед показом, невалидный JSON не ломает UI и не влияет на прогресс.
3. **Режим подсказки отделен от проверки.** `hint` не получает полный `reference`, чтобы снизить риск раскрытия готового ответа.
4. **README/project review не входит в MVP.** Сценарий остается в продуктовой картине, но реализация переносится на отдельный этап после 5.1 и 5.5.
5. **История AI-проверок не входит в MVP.** В MVP показывается только текущий результат без сохранения полного ответа пользователя и AI-feedback.
6. **AI-score не меняет прогресс автоматически.** После feedback пользователь сам выбирает действие.

### Зафиксированные решения

- AI proxy: отдельный Vercel project/serverless endpoint.
- Frontend: хранит только proxy URL и выбранную модель в `#/settings`.
- Key management: DeepSeek API key только в server-side env.
- Default model: `deepseek-v4-pro`.
- Optional mode model split: `deepseek-v4-pro` для `reference_check`, `explain_error`, `sql_review`; `deepseek-v4-flash` для `hint` и `next_step`.
- History: нет в MVP.
- Consent: показывается перед первым AI-запросом и сохраняется локально.
- Data preview: краткий состав отправляемых данных показывается перед отправкой.
- Rate limit: 20 запросов в день и 5 запросов в минуту на IP или anonymous session.
- Invalid AI response: показать безопасную ошибку и не менять прогресс.
- Hint context: без полного `reference`.
- README review: future scope, не MVP.
- Progress: только через явное действие пользователя.
