# Как добавлять практику-тренажёр с датасетами (на любой месяц)

Универсальный промт ниже создаёт **по одной практике на каждый день** месяца там,
где практику реально можно сделать, и сам подбирает формат под навык месяца.

## Как пользоваться
1. Скопируй весь блок ниже.
2. Замени **«МЕСЯЦА N»** в первой строке на нужный месяц (1–7).
3. Вставь в новую сессию Claude Code в этом проекте.

## Что промт делает сам
- Определяет навык месяца из плана (Excel / SQL / Power BI / Python / статистика…).
- Выбирает формат:
  - **SQL** → живая практика с автопроверкой (SQLite в браузере, модуль 5.5).
  - **Остальное** → бизнес-кейс с данными и ключом для самопроверки.
- Датасеты (правило по умолчанию — **гибрид**):
  - **SQL-месяц** → обрезанный реальный **Olist**.
  - **Все прочие месяцы** → реалистичная синтетика по «спеку реализма».

```
Добавь практику-тренажёр для дней МЕСЯЦА N в плане обучения — по одной практике
на каждый день, где её реально можно сделать. Работай в проекте
/Users/a1111/веб/analyst-trainer. Данные плана: learning-plan/data/plan.json.
Кейсы: cases/. Датасеты: cases/datasets/. Не коммить в git без отдельной просьбы.

ШАГ 0 — РАЗБЕРИСЬ С МЕСЯЦЕМ (не пропускать):
- Открой plan.json, найди месяц N: прочитай title/focus/skills и days каждой недели.
- Определи навык месяца (Excel / SQL / Power BI / Python / статистика / и т.д.).
- Изучи по образцу, как устроена практика: cases/sql-*.json + cases/datasets/sql-*.json
  (живой формат), cases/detective-*.json / dashboard-*.json / rca-*.json (бизнес-кейсы),
  ограничения в core/caseValidator.js, словарь тегов core/skillTags.js, реестр модулей
  core/modules.js. Рендер кнопки практики в дне — modules/learning/plan/DayDetail.js
  (функция practiceLaunch: показывает day.caseIds как кнопки → #/module/<m>/case/<id>).

ШАГ 1 — ВЫБЕРИ ФОРМАТ ПРАКТИКИ ПО НАВЫКУ:
A) ЕСТЬ ЖИВОЙ ДВИЖОК (сейчас это только SQL, модуль 5.5 «SQL Investigation»):
   делай ЖИВУЮ практику с автопроверкой:
   - датасет cases/datasets/<caseId>.json:
     { "tables":[ {"name":"...","columns":[{"name":"...","type":"INTEGER|TEXT|..."}],
       "rows":[[...]] } ] }
   - кейс cases/<caseId>.json:
     { "caseId","module":"5.5","schemaVersion":1,"title","difficulty":"basic|intermediate|advanced",
       "skillTags":[...из core/skillTags.js...],
       "payload":{ "scenario","question","datasetPath":"cases/datasets/<caseId>.json",
         "starterSql","subtasks":[{"id","prompt","orderSensitive":false,"referenceSql","explanation"}] } }
   - 3–5 подзадач по нарастанию сложности, ровно под тему дня.
   - referenceSql ОБЯЗАН быть верным для датасета (это эталон автопроверки) — выполни его
     в живом движке и сверь результат.
B) ЖИВОГО ДВИЖКА НЕТ (Excel, Power BI, Python, статистика и пр. — в браузере не запускаются):
   практика = БИЗНЕС-КЕЙС с реальными данными и ключом для самопроверки:
   - предпочитай ПРИВЯЗАТЬ готовый подходящий кейс к дню (модули 5.1 Data Detective,
     5.3 Dashboard Analysis, 5.4 RCA, 5.6 Business Simulator — по 10 кейсов, см. cases/index.json);
   - если подходящего нет — СОЗДАЙ новый кейс по образцу существующего кейса того же
     модуля (копируй payload точь-в-точь, сверяйся с caseValidator.js), вшей в него
     датасет/сценарий, вопросы и в explanation — правильные ответы для самосверки.
   - Авто-проверки SQL-типа тут не будет — это нормально для этих навыков.

ШАГ 2 — ДАТАСЕТЫ (ближе к реальности — это требование, а не пожелание):
- ИСТОЧНИК (правило по умолчанию — ГИБРИД):
  • SQL-месяц → вшей ОБРЕЗАННЫЙ репрезентативный срез реального Olist
    (движок держит данные в памяти, файл ≤5 МБ — не весь датасет, а подвыборка).
  • Все прочие месяцы → реалистичная СИНТЕТИКА под тему дня.
- МАСШТАБ: не 5–10 «ровных» строк, а 150–2000 строк (в пределах ≤5 МБ).
- ЛОКАЛЬНЫЙ КОНТЕКСТ: реальные имена и города КГ/КЗ (Бишкек, Ош, Алматы, Астана,
  Каракол…), валюта сом/тенге, правдоподобные категории товаров, каналы, статусы.
- РЕАЛИСТИЧНЫЙ «БЕСПОРЯДОК» (специально — план про очистку данных): часть NULL,
  дубли строк, разный регистр и опечатки в тексте ('São Paulo' / 'sao paulo' / 'SP'),
  числа, записанные как текст, выбросы (нулевая/огромная сумма), пропуски и
  несогласованные форматы дат.
- РЕАЛИСТИЧНЫЕ РАСПРЕДЕЛЕНИЯ: Парето (немного крупных клиентов дают бóльшую долю
  выручки), сезонность по месяцам, разброс цен — не равномерный шум.
- СВЯЗНОСТЬ ТАБЛИЦ: внешние ключи реально совпадают (заказы ссылаются на существующих
  клиентов/товары), но оставь немного «сирот» — для антиджойнов и проверок качества.
- ФИКСИРОВАННЫЙ СНИМОК: данные не рандомизировать на лету — referenceSql/ключ должны
  давать один и тот же результат. Беспорядок добавляй так, чтобы он был ЧАСТЬЮ задачи,
  а не ломал эталонный ответ.
- СКВОЗНОЙ ДАТАСЕТ: где уместно, держи один датасет на месяц («один датасет — несколько
  сигналов»), от дня к дню усложняя только задачи.

ШАГ 3 — ПРИВЯЖИ К ДНЮ:
- В plan.json у нужного дня (месяц N → неделя → day) добавь day.caseIds = ["<caseId>"]
  (1–2 кейса). Поля day.title/blocks/goal НЕ трогай.

КАКИМ ДНЯМ ПРАКТИКУ НЕ ДЕЛАТЬ:
- «установка/настройка среды», «проектный день», «README/публикация», «экзамен» —
  пропускай (либо привяжи общий повторяющий кейс, если явно просят).

ЖЕЛЕЗНЫЕ ПРАВИЛА:
- Практика по СМЫСЛУ соответствует теме дня, а не «лишь бы привязать».
- Делай по одной, аккуратно, НЕ генерируй пачкой наугад (особенно referenceSql,
  ключи и сами датасеты — проверяй фактически).
- caseId уникальный и говорящий, напр. "sql-m2w2d1-aggregates" или "detective-m1w3-cleaning".
- skillTags только из core/skillTags.js; difficulty только basic/intermediate/advanced;
  module только из тех, у кого hasCases:true (core/modules.js).
- Новый кейс регистрируй в cases/index.json: { caseId, module, path, title, difficulty }.

СБРОС КЭША (после изменений данных, иначе в приложении старое):
- в config.js подними APP_CACHE_VERSION (напр. v1.1 → v1.2),
  запусти: bash scripts/bump-cache-version.sh.

какие датасеты использовал (реальный Olist / синтетика), что проверил.
```
