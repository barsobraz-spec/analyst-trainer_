# Цель проекта

Analyst Trainer — локальное офлайн-приложение (vanilla JS, ES-модули, без сборки,
отдаётся статикой), тренажёр для аналитиков данных.

Текущая продуктовая цель (по PRD): превратить тренажёр в персональную учебную
систему 7-месячного маршрута Data Analyst, объединяющую:
учебный план, режим «Сегодня», чек-лист задач, трекер проектов, карьерный трек
и существующие тренажёрные модули (5.1–5.7) + Learning Analytics (5.8).

Архитектурный принцип, который НЕЛЬЗЯ нарушать: JSON в `learning-plan/data/` и
`cases/` — это контент (только чтение); IndexedDB — единственный источник
пользовательского прогресса.

# Ключевые проблемы

Перечислены в порядке критичности.

1. **Нет версионирования по факту.** В git один коммит, при этом ~13 изменённых и
   несколько untracked файлов (ядро + learning-модуль + `appRoutes.js`,
   `navigation.js`, `modules/learning/plan/DayDetail.js`) висят без коммита. Нет
   контрольных точек и отката — высокий риск потери работы и «внезапных» поломок.
2. **Нет автотестов / линта / CI.** Контроль качества — только ручные `?smoke=`
   проверки в консоли. Регрессии не отлавливаются (вероятный источник жалобы
   «кнопки не работают» на отдельном экране).
3. **Жалоба «кнопки не работают» не воспроизводится глобально.** При запуске через
   локальный HTTP-сервер навигация, аккордеоны, ссылки и кнопки экранов работают,
   ошибок в консоли нет. Наиболее вероятная реальная причина — запуск через
   `file://` (тогда ES-модули/JSON/воркеры блокируются и приложение мертво) либо
   конкретный экран в процессе незакоммиченного рефакторинга.
4. **Контент грузится целиком в память.** `loadLearningContent()` тянет все 5 JSON
   разом (`practiceContent.json` ≈ 1 МБ, `plan.json` ≈ 234 КБ); `loadAllCases()`
   грузит все файлы кейсов параллельно. Не масштабируется на 1000+ уроков /
   10000+ задач.
5. **Ручные cache-bust строки** (`?v=topic-tasks-1`) расставлены вручную и
   непоследовательно (`index.html`, `main.js`, `appRoutes.js`) — хрупко, легко
   забыть, старый кэш ломает экран.
6. **Жёстко зашитая таксономия модулей** (`5.1`–`5.8`) как магические строки в
   нескольких местах. Добавление нового типа контента/языка требует правок во
   многих файлах.
7. **Нет слоя i18n** — все строки UI зашиты по-русски в коде. Мультиязычность
   потребует сквозной переработки.
8. **Монолитные файлы** — `db.js` (998 строк), `styles.css` (5075 строк / 151 КБ
   одним файлом), ряд View (400–530 строк). Растут без декомпозиции.
9. **Шум репозитория** — `.DS_Store` повсюду; рядом второй проект
   (`my-data-analyst-lab`) создаёт риск правки не того кода.

# План работ

Этапы упорядочены так, чтобы сначала закрыть риск потери работы и поломок
(гигиена + страховка), затем убрать хрупкость, и только потом — масштабирование и
новые возможности. Каждый этап самостоятелен и не ломает рабочий функционал.

## Этап 4: Устранение хрупкости кэша (версионирование ассетов)

* [x] Применить её единообразно ко всем версионируемым импортам/стилям
      — index.html (styles.css, main.js), main.js (appRoutes.js), core/appRoutes.js (TasksView.js) — все ?v=v1.0
* [x] Документировать правило: версия меняется в одном месте
      — комментарий в config.js + README (раздел «Техническое замечание о кэше»)

Изменяемые файлы:

* `index.html`
* `main.js`
* `core/appRoutes.js`

Не трогать:

* логику модулей и роутера
* структуру данных

Критерии готовности:

* нет ручных рассинхронизированных `?v=topic-tasks-1`
* смена версии в одном месте обновляет все ассеты
* приложение открывается без ошибок импорта

Риски:

* неверный путь с версией → 404 модуля и белый экран. Проверять все импорты
* кэш браузера при тестировании — проверять в режиме hard reload

Проверка после выполнения:

* автоматически: smoke-прогон (Этап 3) зелёный

* `docs/` (архитектурная заметка)

Не трогать:

* пользовательские данные и их миграции
* видимые тексты не должны измениться (язык по умолчанию — русский)

Критерии готовности:

* UI выглядит идентично, но строки идут через словарь
* существующие проверки работают через единый интерфейс grader
* есть документ «как добавить язык / тип контента / проверяющего»

Риски:

* массовый вынос строк — риск пропустить/исказить текст. Делать по экранам с
  визуальной сверкой
* преждевременная абстракция grader — держать минимально достаточной

Проверка после выполнения:

* вручную: визуально сверить ключевые экраны — тексты не изменились
* автоматически: smoke-прогон; проверка отсутствия «потерянных» ключей словаря
* сценарий: добавить тестовый язык-заглушку → переключение работает; убрать

# Что категорически запрещено

* не менять существующие ID (модулей `5.1`–`5.8`, кейсов, задач, ключей хранилища)
* не ломать существующие данные пользователя в IndexedDB
* не удалять рабочий функционал
* не менять структуру данных без миграции (для IndexedDB — только через механизм
  `MIGRATIONS` в `core/db.js` с поднятием `APP_SCHEMA_VERSION`)
* не создавать временные решения и заглушки «на потом» в проде
* не переносить пользовательский прогресс в JSON (JSON — только контент)
* не запускать приложение через `file://` при проверке (только локальный HTTP)

# Future Architecture

Архитектура, рассчитанная на рост до 1000+ уроков, 10000+ задач, мобильное
приложение, AI-проверку ответов, мультиязычность и новые типы контента.

## 1. Слои и границы

* **Контент** (`learning-plan/data`, `cases`) — неизменяемый источник, версионируемый,
  только чтение. Хранится как **индекс + чанки**, а не монолитные файлы.
* **Состояние пользователя** — только IndexedDB, единственный источник прогресса,
  изменяется через миграции. Эта граница (контент ≠ состояние) — нерушима.
* **Прикладной слой** (`core`) — роутер, загрузчики, хранилище, проверяющие.
* **Представления** (`modules`) — экраны, читают контент и состояние, не знают про
  способ их хранения.

## 2. Масштабирование контента (1000+ уроков, 10000+ задач)

* Контент разбит на **манифест/индекс + ленивые чанки** (по месяцу/теме/модулю).
* Загрузка **по требованию экрана**, кэш в памяти с ограничением размера.
* Длинные списки — **виртуализация/пагинация**, а не полный рендер.
* Проверка целостности ссылок контента (`taskIds`/`caseIds`) — отдельный скрипт.

## 3. Расширяемые типы контента

* Единый **реестр типов**: `{ id, title, schema, View, grader }` в одном месте.
* Новый тип = новая запись в реестре + новый View + (опц.) новый grader.
  Никаких правок в роутере, хосте кейсов и валидаторе.

## 4. AI-проверка ответов

* Интерфейс **`Grader`** с методом проверки ответа → результат+обратная связь.
* Текущие проверки (`sqlComparator`, самооценка) — первые реализации этого
  интерфейса.
* AI-grader — ещё одна реализация за тем же интерфейсом; **офлайн-режим всегда
  имеет локальный fallback** (детерминированную проверку), AI — опциональный слой.

## 5. Мультиязычность

* Все пользовательские строки — через **централизованный словарь** (ключ → перевод),
  язык по умолчанию — русский.
* Контент-чанки могут существовать в языковых вариантах под общим id.
* Переключение языка не трогает прогресс и id.

## 6. Мобильное приложение

* Ядро (`core`) и контентные контракты не зависят от DOM-специфики настолько,
  насколько возможно, чтобы переиспользоваться в обёртке (PWA / нативная оболочка).
* Хранилище абстрагировано (сейчас IndexedDB) — мобильная обёртка может подставить
  свой адаптер за тем же интерфейсом.
* Адаптивная вёрстка и нижняя навигация уже есть — развивать как PWA-кандидат.

## 7. Качество и устойчивость

* Smoke-набор по всем роутам + проверка обязательных интерактивных элементов.
* Единая точка версионирования ассетов (кэш-бастинг) вместо ручных меток.
* Git с осмысленной историей и контрольными точками перед каждым этапом.
* Миграции данных — только аддитивные, со своим тестом до релиза.
